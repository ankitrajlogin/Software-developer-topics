# 🌐 Web Sessions, Spring Security & CSRF Protection — Complete Notes

---

## 1️⃣ HTTP Is Stateless — Why Sessions Exist

HTTP protocol does **not remember users**.
Each request is independent.

Without session:

* Server cannot identify the same user across requests
* Login, cart, preferences are impossible

👉 **Session adds memory to HTTP**

---

## 2️⃣ What Is an HTTP Session?

An **HTTP Session** is a server-side storage mechanism that:

* Identifies a browser/user
* Stores temporary data
* Is linked using a **session ID**

### How Session Is Created

1. User sends first request
2. Server creates a session
3. Server generates a session ID (e.g. `JSESSIONID`)
4. Session ID is sent to browser as a cookie
5. Browser sends this cookie on every next request

📌 No login is required to create a session

---

## 3️⃣ Session Without Login (Very Important)

Even if:

* No authentication
* No username/password

Session still exists.

### Why?

Because:

* Session identifies **browser**, not **user identity**
* Login only stores user data *inside* session

---

## 4️⃣ Benefits of Session

Sessions allow:

* User identification
* Login persistence
* Shopping cart
* Multi-step forms
* OTP flows
* Better performance (cached data)

Sessions make web apps **stateful over stateless HTTP**.

---

## 5️⃣ Session vs Cookies (Important Difference)

| Aspect    | Session | Cookie    |
| --------- | ------- | --------- |
| Stored    | Server  | Browser   |
| Size      | Large   | Small     |
| Security  | Safer   | Less safe |
| Auto-sent | Via ID  | Yes       |

Cookies usually store **session ID**, not data.

---

## 6️⃣ Spring Security — What It Does

Spring Security provides:

* Authentication (Who are you?)
* Authorization (Are you allowed?)
* Session management
* CSRF protection
* Password encryption
* Security filters

Everything passes through **Security Filter Chain** before controller.

---

## 7️⃣ Authentication vs Authorization (Status Codes)

| Scenario                    | HTTP Code        |
| --------------------------- | ---------------- |
| Not logged in               | 401 Unauthorized |
| Logged in but no permission | 403 Forbidden    |

Memory trick:

* **401 → Who are you?**
* **403 → You are known, but blocked**

---

## 8️⃣ What Is CSRF?

**CSRF = Cross-Site Request Forgery**

It happens when:

* User is logged in
* Browser has valid cookies
* Attacker tricks browser into sending request

Browser automatically sends cookies.
Server thinks request is genuine.

---

## 9️⃣ Why Login Alone Cannot Stop CSRF

Because:

* Browser auto-sends cookies
* Attacker never needs password
* Server only checks session

👉 Result: unauthorized action

---

## 🔟 CSRF Token — Core Idea

Server requires **two proofs**:

1. Cookie (session proof)
2. CSRF Token (user intent proof)

Attacker can send cookies ❌
Attacker cannot send CSRF token ❌

---

## 1️⃣1️⃣ Where CSRF Token Is Stored

### Server Side

* Stored inside **HTTP Session**

### Client Side (Delivery)

* Hidden form field
* Meta tag
* HTTP header

Token is generated by server.
Browser never generates token.

---

## 1️⃣2️⃣ Is CSRF Token Per Tab?

❌ No

✅ CSRF token is **per session**

All tabs in same browser:

* Share same session
* Share same CSRF token

Different browser / incognito:

* New session
* New CSRF token

---

## 1️⃣3️⃣ Same-Origin Policy (SOP)

Browser security rule:

> One website cannot read data of another website

Blocks access to:

* HTML
* DOM
* JavaScript
* Cookies
* CSRF token

This rule protects CSRF tokens.

---

## 1️⃣4️⃣ Why Evil Website Can Send Cookies

Because:

* Cookies are sent automatically by browser
* Browser matches domain
* JavaScript is not involved

Evil site:

* ❌ Cannot read cookies
* ❌ Cannot modify cookies
* ✅ Browser sends cookies

---

## 1️⃣5️⃣ Why Evil Website Cannot Send CSRF Token

Because:

* CSRF token is stored in DOM/HTML
* Reading token requires same-origin access
* SOP blocks cross-site reading

Attacker cannot:

* Read token
* Guess token
* Reproduce token

---

## 1️⃣6️⃣ Why Incognito Mode Helps

Incognito:

* Uses separate cookie store
* Has no existing session

So:

* Cookies not sent
* CSRF attack fails

⚠️ If you log in inside incognito → cookies exist again

---

## 1️⃣7️⃣ When CSRF Protection Fails

Only when **XSS exists**.

XSS allows attacker to:

* Run JS on same site
* Read CSRF token
* Bypass protection

👉 XSS is more dangerous than CSRF.

---

## 1️⃣8️⃣ CSRF vs XSS

| CSRF            | XSS                   |
| --------------- | --------------------- |
| Forces actions  | Steals data           |
| Cross-site      | Same-site             |
| Fixed by tokens | Fixed by sanitization |

---

## 1️⃣9️⃣ When to Enable CSRF

| App Type                  | CSRF      |
| ------------------------- | --------- |
| Web app (session + forms) | ✅ Enable  |
| Banking / ecommerce       | ✅ Must    |
| REST API (JWT)            | ❌ Disable |
| Mobile backend            | ❌ Disable |

---

## 2️⃣0️⃣ Key Interview One-Liners

* Session exists even without login
* CSRF token is stored in session
* CSRF token is per session, not per tab
* Browser sends cookies automatically
* Attacker cannot read CSRF token due to SOP
* CSRF protects user intent

---

## 2️⃣1️⃣ Final Memory Model

* **Cookie** → identity proof (browser)
* **CSRF Token** → intent proof (user)
* **Session** → server memory
* **Spring Security** → enforcement layer

---

## ✅ End of Notes

These concepts form the **foundation of secure web applications**.
